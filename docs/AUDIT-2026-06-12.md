## Verdict

**Healthy early-stage project. Ship-capable for single-process use, with caveats.**

The code is clean, well-typed (`strict: true`), and unusually well-tested for a 0.1.x tool (312 passing tests across 19 files). Defensive engineering is everywhere: atomic state writes with checksums, path-traversal guards, graceful shutdown, budget pre-flight checks, stale-lock recovery.

The findings below are not "it's broken" — they are the gaps that will bite as the tool meets real `.pen` files, real CSS conventions, and concurrent processes. Two of them (P1-A, P1-B) are worth resolving before promoting this past "works on my setup."

## P1 — Resolve before wider release

### P1-A. The `.pen` "encrypted file" guardrail contradicts the code that reads it

`CLAUDE.md` and the project conventions state, repeatedly and emphatically:

> `.pen` file contents are encrypted — always use Pencil MCP tools, never `Read`/`Grep` on `.pen` files.

But the core read path does exactly that:

- `src/pen-reader.ts` → `JsonPenReader.readSnapshot()` calls `readFile(penFile, "utf-8")`
- `src/pen-snapshot.ts` → `snapshotPenFile()` calls `JSON.parse(raw)`
- `src/pen-to-code.ts` and `src/sync-engine.ts` both `readFile(mapping.penFile)` then `JSON.parse`

If real `.pen` files are genuinely encrypted, every pen-to-code sync and every post-sync snapshot will fail at `JSON.parse` → `snapshot === null` → `"Pen file contains invalid JSON"`. The entire snapshot/diff/fast-path engine is built on the assumption that `.pen` is plaintext JSON.

**This is the single most important question for whether the tool works on real Pencil.dev files.** Either:
1. The files in scope are plaintext JSON exports/snapshots (then the "encrypted, never Read" guardrail is wrong and misleading), or
2. Real `.pen` files are encrypted (then the JSON-based reader is dead code against real inputs and the whole pen-to-code path needs to go through the Pencil MCP, not `readFile`).

**Action:** Resolve the contradiction explicitly. Document which `.pen` shape the `JsonPenReader` targets, and either correct the guardrail or route reads through the MCP. The `PenReader` interface already exists as the seam for an `McpPenReader` — that abstraction looks designed for exactly this, but only the `Json` implementation ships.

### P1-B. Budget enforcement can be silently bypassed

Budget is enforced two ways, and both have holes:

1. **Pre-flight estimate** (`sync-engine.ts:checkBudget`) only estimates **input** tokens (`estimateInputTokens` = `prompt.length / 4`). Output tokens — the expensive side (`$15–75`/M vs `$3–15`/M input) — are never pre-estimated. A single generation can overshoot `maxBudgetUsd` by a full response.

2. **Post-hoc tracking** (`trackSpend`) depends entirely on `parseTokenUsage(stderr)`, which scrapes the Claude CLI `--verbose` stderr with loose regex (`/input[_\s]tokens["\s:]*(\d+)/i`). If the CLI changes its output format, `parseTokenUsage` returns `undefined` → `trackSpend` no-ops → `cumulativeSpendUsd` stays `0` → the budget is **never enforced** across calls. There is no fallback estimate from prompt+response length when parsing fails.

Net effect: a format drift in the upstream CLI turns `maxBudgetUsd` from a hard cap into a no-op, and nothing logs that enforcement stopped working.

**Action:**
- Add an output-token estimate to the pre-flight check (even a conservative multiple of input).
- When `parseTokenUsage` returns `undefined`, fall back to a length-based estimate of both input and output rather than recording `0`, and `log.warn` that token parsing failed so the bypass is visible.

## P2 — Should fix

### P2-A. `LockManager` is in-memory only, but is described as file-based

`CLAUDE.md` project structure: *"lock-manager.ts File-based locking"*. The prior `docs/REVIEW.md` repeats this. The actual implementation (`src/lock-manager.ts`) is an in-process `Map<string, boolean>` — it only coordinates within **one** Node process.

Consequence: running `pencil-sync watch` and `pencil-sync sync` simultaneously (or two watchers) on the same mapping gives each its own lock map. They can both edit code files and write `.pencil-sync-state.json` concurrently → torn state, lost edits. The atomic-rename in `StateStore.save()` prevents a *corrupt* file but not a *last-writer-wins* race.

**Action:** Either implement real file-based locking (lockfile with PID + mtime, which the `STALE_LOCK_MS` machinery already anticipates) or correct the docs to state locking is single-process only and add a startup guard against a second process on the same state file.

### P2-B. Two different glob engines — watch vs. hash

- The **watcher** (`watcher.ts`) matches files using chokidar's built-in globbing (picomatch-class semantics: braces, extglobs, character classes).
- **State hashing / conflict detection** (`state-store.ts:globToRegex`) uses a hand-rolled glob→regex that supports only `*`, `**`, `**/`, `?`, and literal `.`. It does **not** support `{a,b}` brace expansion, `[abc]` classes, or `?(...)` extglobs.

A `codeGlobs` entry like `components/**/*.{tsx,jsx}` will be watched correctly by chokidar but will **never match** in `globToRegex` → changes are detected by the watcher but produce "no diff," or files are silently excluded from the conflict baseline. Divergence between "what we watch" and "what we hash" is a correctness trap.

**Action:** Use one matching engine for both. Reusing `picomatch` (already in the tree transitively) for `collectFiles` would unify semantics. At minimum, validate `codeGlobs` at config-load time and reject patterns `globToRegex` can't represent.

### P2-C. Color fast-path silently assumes one specific CSS convention

`pen-to-code.ts:applyFillChanges` only matches CSS variables in the form `--color-NAME: R G B;` (space-separated RGB channels, Tailwind v4 style). If the project uses:
- hex (`--color-x: #224846;`),
- `rgb(...)` / `hsl(...)` functional notation, or
- any prefix other than `--color-`,

…the regex finds nothing, records `"old RGB not found"` as a **non-blocking warning**, and the sync returns `success: true` with `0` files changed. A user changing a color in Pencil sees "sync complete" and no change in their app, with the reason buried at warn level.

**Action:** Detect the project's color format (or make it configurable), and when a fill diff matches **zero** declarations, surface it as a visible failure/skip in the `SyncResult`, not just a log line.

### P2-D. Two divergent path-traversal implementations, one POSIX-only

- `utils.ts:validatePathWithin` uses `path.relative` + `sep` — cross-platform correct.
- `config.ts:validatePathWithinDirectory` uses `normalizedResolved.startsWith(normalizedAllowed + "/")` — hardcodes the POSIX `/` separator. On Windows, `path.resolve` yields `\` separators, so this check would throw false-positive traversal errors on legitimate paths.

Both also normalize but neither resolves symlinks (`realpath`), so a symlink inside `codeDir` pointing outside it still passes. Low risk given config is user-owned, but worth noting.

**Action:** Consolidate on the `utils.ts` `relative`/`sep` approach and delete the duplicate. Consider `realpath` if untrusted configs ever become a concern.

### P2-E. `collectFiles` ignore-list drifts from the watcher's

`state-store.ts:collectFiles` hardcodes skipping `node_modules`, `.git`, `dist`. The watcher (`watcher.ts`) additionally ignores `.next`. So a glob that matches `.next/**` is excluded while watching but **hashed** during conflict detection → phantom "code changed" events from build output.

**Action:** Share a single ignore list between the watcher and `collectFiles`.

## P3 — Minor / hygiene

- **No retries outside MCP mode.** `runClaude` only retries when `mcpConfigPath` is set (`claude-runner.ts:185`). Transient failures (network blips) in normal mode get zero retries. Reasonable, but document it.
- **Stale model pricing.** `MODEL_PRICING` (`claude-runner.ts`) lists `claude-opus-4-6` / `claude-sonnet-4-6`; the current Opus generation is 4.8. Any model not in the map silently falls back to **Sonnet** pricing, skewing budget math. Consider warning on unknown models.
- **No ESLint.** `strict: true` catches types, but there's no lint layer for style/dead-code/`no-floating-promises`. Several `async` event handlers in `watcher.ts` are fire-and-forget; a lint rule would flag intent.
- **No `npm audit` gate in CI.** The 4 advisories below went unnoticed. A non-blocking `npm audit --omit=dev` step would catch *runtime* advisories early.
- **No coverage report.** 312 tests is strong, but `vitest.config.ts` has no coverage thresholds, so regressions in coverage are invisible.
- **`askUser` reads global `process.stdin`** inside the engine. Fine for CLI, but couples the engine to a TTY; the non-TTY guard correctly defaults to skip.

## Dependency advisories (`npm audit`)

| Package | Severity | Path | Shipped to users? |
|---|---|---|---|
| `vitest` (UI server RCE) | **Critical** | `vitest` | ❌ dev-only |
| `vite` (path traversal / file read) | High | `vitest` → `vite` | ❌ dev-only |
| `picomatch` (ReDoS / glob injection) | High | `vitest` → `tinyglobby`/`vite`/`fdir` | ❌ dev-only |
| `postcss` (XSS in stringify) | Moderate | `vitest` → `vite` | ❌ dev-only |

**All four are confined to the `vitest` dev-dependency tree.** The published npm package ships only `dist/`, `prompts/`, `README`, `LICENSE` (per `package.json#files`), and `chokidar` 4 no longer depends on `picomatch`, so **none of these reach consumers**. The vitest-UI critical only triggers when the UI server is running, which this project never starts.

**Action (low urgency):** `npm audit fix` to bump the vitest chain; add a CI audit step scoped to runtime deps.

## What's done well (keep)

- **Atomic, checksummed, backed-up state** (`state-store.ts`): tmp-write + rename, SHA-256 integrity check, orphaned-tmp cleanup, `.backup` copy. Textbook.
- **Graceful shutdown** (`shutdown.ts` + signal/rejection/exception handlers in `index.ts`) with timeouts — and it's tested.
- **Echo-suppression + grace period** in `LockManager` to prevent pen↔code ping-pong loops. Subtle problem, handled deliberately.
- **Prototype-pollution-safe config merge** (`safeMerge` blocks `__proto__`/`constructor`/`prototype`).
- **JSONC parsing that preserves string contents** before stripping comments — easy to get wrong, done right.
- **Dry-run support** end-to-end with dedicated tests.
- **CI security posture**: action SHAs pinned, ShipGuard version pinned, dual Node matrix.
- **Test breadth**: lock state machine, state recovery, partial success, watcher resilience, shutdown — the hard-to-test failure paths are covered.

## Recommended next actions (priority order)

1. **Settle the `.pen` encryption question (P1-A).** Everything downstream depends on it. Document the real `.pen` shape; wire an `McpPenReader` if files are encrypted.
2. **Harden budget enforcement (P1-B).** Estimate output tokens pre-flight; fall back + warn when `parseTokenUsage` fails instead of silently recording `0`.
3. **Unify glob engines (P2-B)** and the ignore list (P2-E) so "watched" == "hashed."
4. **Decide LockManager scope (P2-A):** real file lock, or document single-process and guard against a second process.
5. **Make color fast-path failures visible (P2-C)** and consolidate the path-traversal helpers (P2-D).
6. **Housekeeping:** `npm audit fix`, add ESLint + a runtime-scoped `npm audit` CI step, add coverage thresholds, refresh `MODEL_PRICING`.
